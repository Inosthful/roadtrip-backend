### Concepts clés expliqués à l'étudiant

**1. trip_participants (Many-to-Many)**
- Un user peut participer à plusieurs trips
- Un trip peut avoir plusieurs participants
- Table pivot nécessaire pour cette relation
- Bonus : roles et invitation_status

**2. is_locked dans stops**
- Ordre FIXE pour l'algorithme d'optimisation
- L'algo peut réorganiser les stops non-locked
- Les stops locked gardent leur position

**3. expense_splits (Système de balance)**
- Calcul de "qui doit combien à qui"
- Principe créditeur/débiteur
- is_paid track les remboursements

### 2. Logique métier avancée implémentée ✅

#### **A. Création automatique des expense_splits (méthode store)**

Quand une dépense est créée, le système :
1. Récupère tous les participants du voyage
2. Calcule le montant par personne (`expense.amount / nbParticipants`)
3. Crée automatiquement un `ExpenseSplit` pour chaque participant
4. Marque `isPaid: true` pour le payeur, `false` pour les autres

**Code clé :**
```typescript
await trip.load('participants')
const nbParticipants = trip.participants.length
const montantParPersonne = expense.amount / nbParticipants

const splitsData = trip.participants.map((participant) => ({
  expenseId: expense.id,
  userId: participant.id,
  amount: montantParPersonne,
  isPaid: participant.id === user.id,  // Opérateur ternaire compact
}))

await ExpenseSplit.createMany(splitsData)  // Création en masse (1 seule requête)
```

#### **B. Algorithme de calcul de balance (méthode balance)**

Endpoint `/trips/:tripId/balance` qui calcule **qui doit combien à qui** 💡

**Algorithme complet :**
1. Récupérer toutes les dépenses du voyage avec leurs splits
2. Créer un objet `balances` pour accumuler les données (clé = userId)
3. Pour chaque expense :
   - Ajouter `expense.amount` au `paid` du payeur
   - Parcourir les splits et ajouter `split.amount` au `owes` de chaque participant
4. Calculer `balance = paid - owes` pour chaque user
5. Transformer l'objet en tableau avec `Object.values()`

**Structure de données retournée :**
```json
{
  "tripId": 1,
  "tripTitle": "Voyage en Italie",
  "balances": [
    { "userId": 5, "fullName": "Alice", "paid": 90, "owes": 50, "balance": 40 },
    { "userId": 8, "fullName": "Bob", "paid": 60, "owes": 50, "balance": 10 },
    { "userId": 12, "fullName": "Charlie", "paid": 0, "owes": 50, "balance": -50 }
  ]
}
```

**Interprétation :**
- Alice : `balance = +40€` → On lui doit 40€ (créditeur)
- Bob : `balance = +10€` → On lui doit 10€ (créditeur)
- Charlie : `balance = -50€` → Il doit 50€ (débiteur)

### 3. Concepts avancés appris 🎓

#### **A. Lazy Loading vs Eager Loading**
```typescript
// Eager Loading (.preload sur une query - 1 requête optimisée)
await Trip.query().preload('participants').findOrFail(id)

// Lazy Loading (.load sur un model déjà chargé - 2 requêtes séparées)
const trip = await Trip.findOrFail(id)
await trip.load('participants')  // ⚠️ Asynchrone, nécessite await !
```

**Quand utiliser quoi ?**
- `.preload()` : Quand on charge le model ET les relations en même temps
- `.load()` : Quand on a déjà le model et qu'on veut charger les relations après

#### **B. Pattern Accumulateur**
```typescript
balances[userId].paid += amount  // ✅ Accumule les valeurs
balances[userId].paid = amount   // ❌ Écrase (perd les valeurs précédentes)
```

#### **C. Structures de données : Objet vs Tableau**
**Pourquoi un objet pour `balances` ?**
```typescript
const balances: Record<number, {...}> = {}  // Objet avec clés numériques
balances[5] = {...}  // Accès direct O(1)
```

**Alternative tableau (MOINS PERFORMANT) :**
```typescript
const balances = []
const user = balances.find(b => b.userId === 5)  // Recherche O(n)
```

**Transformation finale :**
```typescript
Object.values(balances)  // Objet → Tableau pour l'API JSON
```

#### **D. TypeScript `Record<K, V>`**
```typescript
Record<number, { userId: number; fullName: string; ... }>
// = Objet avec clés de type number et valeurs de type {...}
```

#### **E. Création en masse avec `.createMany()`**
```typescript
// ❌ LENT : N requêtes SQL
for (const split of splitsData) {
  await ExpenseSplit.create(split)
}

// ✅ RAPIDE : 1 seule requête SQL
await ExpenseSplit.createMany(splitsData)
```

#### **F. Boucles en TypeScript**
```typescript
for (const expense of expenses)    // Itère sur les VALEURS d'un tableau
for (const userId in balances)     // Itère sur les CLÉS d'un objet
```

#### **G. Lazy Initialization**
```typescript
if (!balances[userId]) {
  balances[userId] = { paid: 0, owes: 0, ... }  // Créer seulement si nécessaire
}
balances[userId].paid += amount  // Puis accumuler
```

### 4. Complexité algorithmique 📊

**Algorithme de balance :**
```typescript
for (const expense of expenses) {           // O(n) - n dépenses
  for (const split of expense.splits) {     // O(m) - m splits par dépense
    // Traitement O(1)
  }
}
// Complexité totale : O(n × m)
```

**Optimisation spatiale :**
- Objet `balances` : O(p) où p = nombre de participants (généralement petit)
- Accès O(1) grâce aux clés d'objet

headers: {

signal: AbortSignal.timeout(5000)
